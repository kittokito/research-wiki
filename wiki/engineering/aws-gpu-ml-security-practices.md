# AWS上でGPU分散学習・ML開発をする自分向け注意事項まとめ

## 前提

自分は一般的なWebサービスの本番運用担当ではないが、以下のような業務を行う。

- AWS上でのGPUインスタンス利用
- 分散学習
- IAM / S3 / EC2 / SSH 接続の運用
- チームでの共同開発
- Web由来データや外部データの学習利用
- モデル、チェックポイント、ログ、評価結果の保存・共有

この前提では、気をつけるべき点は **Webアプリの脆弱性そのもの** よりも、以下に集中する。

- 権限の出しすぎ
- 学習データ・ログ・成果物からの情報漏洩
- SSHやノード管理の雑な運用
- 秘密情報の混入
- データ利用上の法務・倫理・機密の問題
- チーム運用時の境界の曖昧さ
- コスト事故と監査不能

---

## 1. 最重要の考え方

まず一番大事なのは、次の認識を持つこと。

### 1-1. 学習基盤は「ただの計算環境」ではない

学習基盤には、以下のような価値の高いものが集まる。

- 生データ
- 前処理済みデータ
- 特徴量
- モデル重み
- LoRA / adapter
- 評価ログ
- エラーログ
- チームの設定ファイル
- APIキーや認証情報
- 社外秘の仕様やプロンプト

つまり、学習基盤は実質的に **機密データの集積所** になりやすい。

### 1-2. 「サービスを公開していないから安全」は誤り

自分がWebアプリを公開していなくても、以下の事故は普通に起こりうる。

- S3の誤公開
- IAMロールの権限過多
- SSH鍵の流出
- ノード侵害による認証情報窃取
- ログ共有による機密流出
- 学習データに個人情報や秘密情報が混入
- チェックポイントや出力にデータが残留

### 1-3. 一番怖いのは「便利だから」で雑に広げた運用

ML開発ではスピード優先になりやすく、次のような判断をしがち。

- とりあえず強いIAMを付ける
- 一時的にS3を広く開ける
- SSHで直接入る
- `.env` や設定ファイルに鍵を置く
- 学習ノードから何でもアクセスできるようにする
- チームで共通鍵を使う
- ログに全部出す

この「一時対応」がそのまま常態化するのが危険。

---

## 2. IAM: 権限を広げすぎない

### 2-1. 一番やってはいけないこと

- 人間ユーザーに常時 `AdministratorAccess`
- 学習ノードのインスタンスロールに `s3:*`
- どのバケットにもアクセスできる共通ロール
- 実験のために付けた権限を消さない
- IAM変更権限まで開いてしまう

### 2-2. なぜ危険か

権限が広いと、以下のどれが起きても被害が大きくなる。

- 自分の誤操作
- チームメンバーの誤操作
- バグ
- ノード侵害
- 外部ツールやライブラリ由来の不正動作

つまり、権限過多は「事故が起きた後の被害半径」を大きくする。

### 2-3. 自分が意識すべき設計

#### 人間用権限と実行環境用権限を分ける

- 自分の操作用ロール
- 学習ジョブ用ロール
- 推論ジョブ用ロール
- データ収集ジョブ用ロール
- CI/CD用ロール
- 管理用ロール

を分ける。

#### 必要最小限をprefix単位で付ける

例えばS3なら、バケット全体ではなく、

- `datasets/project-a/`
- `checkpoints/exp-001/`
- `logs/team-x/`

のように範囲を絞る。

#### 「読む」と「書く」と「消す」を分ける

- 読み取りだけ必要
- 書き込みだけ必要
- 削除は不要

ということが多い。削除権限は特に慎重にする。

### 2-4. 自分へのルール

- 常設の強権限を持たない
- 「一時的に」広げた権限は必ず戻す
- バケット単位ではなくprefix単位で考える
- 人間とインスタンスの権限を混ぜない
- IAM変更権限は本当に必要な人だけに絞る

---

## 3. S3: データ・モデル・ログの境界を曖昧にしない

### 3-1. S3はただの保存先ではない

S3には次のものが置かれやすい。

- 元データ
- 前処理後データ
- 学習用キャッシュ
- tokenizerや中間成果物
- モデル重み
- checkpoint
- eval結果
- サンプル出力
- 失敗ログ
- notebook出力

これらは全部、漏れて困る可能性がある。

### 3-2. ありがちな事故

- 誤って公開設定にする
- presigned URL を雑に共有する
- 機微データと公開可能成果物を同じ場所に置く
- 外部共有用に置いたものから別のデータも見えてしまう
- 誤削除
- 上書き
- 別案件のデータが混ざる

### 3-3. 特にMLで怖いこと

MLでは、原データだけでなく **派生物も危険**。

#### 危険な派生物の例

- embedding cache
- eval sample
- bad case集
- 生成例
- ログ中の入力文
- fine-tune dataの断片
- checkpointに残るデータ起因の情報

つまり、見た目が「モデル成果物」でも中身に機密が残っていることがある。

### 3-4. 自分がやるべき整理

最低でも保存先を分ける。

- 生データ
- 前処理済みデータ
- 学習中間生成物
- モデル成果物
- チーム共有用成果物
- 外部共有用成果物
- ログ / 監査用

できればprefix分離ではなく、用途ごとにバケットやアカウント境界も検討する。

### 3-5. 自分へのルール

- S3を一時公開しない
- 外部共有用と内部保管用を分ける
- 生データと成果物を同居させない
- 学習ログに入力データ断片が残っていないか意識する
- checkpointや出力は「漏れて困る前提」で扱う

---

## 4. EC2 / GPUノード / SSH: 「入れる」ことより「安全に管理できる」ことを重視する

### 4-1. SSH中心運用の危険

GPUノード運用ではSSHが便利だが、SSH前提にすると以下が起きやすい。

- 22番ポート開放
- 許可元が広すぎる
- 鍵の使い回し
- 誰がどのノードに入ったか追いにくい
- 退職者・外部メンバーの鍵が残る
- head node が踏み台化する

### 4-2. ノードは侵害されうる前提で考える

学習ノードは以下の理由で意外と危ない。

- 実験コードが頻繁に変わる
- 外部ライブラリを多用する
- notebookや補助スクリプトが雑になりやすい
- root権限やsudoを使いがち
- 研究用にいろいろなソフトが入る

だから、「ノードに入られても被害を局所化できるか」を考えるべき。

### 4-3. 特に気をつけること

#### head node に権限を集めすぎない

head node 経由で全部できる状態は危険。

#### compute node に不要な外向き通信を許さない

学習に不要な外部通信は減らす。

#### ローカルディスク残骸

- `/tmp`
- NVMe
- キャッシュ
- downloaded dataset
- ssh known_hosts
- notebook出力

などに機密が残ることがある。

#### AMI / Dockerイメージに秘密を焼き込まない

- APIキー
- SSH鍵
- `.env`
- 認証トークン
- Cookie
- 接続先URL

などをイメージに含めない。

### 4-4. 自分へのルール

- SSHは必要最小限
- head node を強権限の集中点にしない
- worker node に不要な権限を載せない
- ノード再利用時はローカル残骸を意識する
- イメージやスナップショットに秘密を含めない

---

## 5. Secrets管理: コードよりログと設定ファイルが危ない

### 5-1. 漏れやすい場所

秘密情報はコードに直書きしなければ安全、ではない。実際は次の場所に漏れやすい。

- `.env`
- shell history
- notebook
- Hydra/YAML設定
- 実験設定のdump
- wandb / MLflow のconfig
- ログ出力
- 例外メッセージ
- Slack貼り付け
- READMEやメモ

### 5-2. 具体的に危険なもの

- OpenAI / Anthropic などのAPIキー
- Hugging Face token
- GitHub token
- Webhook URL
- DB接続情報
- 外部サイトの認証Cookie
- SSH秘密鍵
- SaaSのアクセストークン

### 5-3. ML開発で危ない理由

モデル開発では、
- 設定共有
- 再現性のためのconfig保存
- 実験管理
- notebookの共有
- CLIコマンドのコピペ

が多いため、**秘密が「開発の便利さ」に埋もれて漏れやすい**。

### 5-4. 自分へのルール

- 秘密情報は設定ファイルに直書きしない
- config dump 前提のツールにはマスキングを入れる
- notebook共有前に token を確認する
- shellコマンドに機密を直接書かない
- ログにHTTPヘッダや認証情報を出さない
- 漏れた前提で再発行できる体制を意識する

---

## 6. ログ・実験管理・可観測性: ログは宝でもあり漏洩源でもある

### 6-1. ログに残りやすい危険情報

- 学習データのサンプル
- プロンプト全文
- モデル出力
- 失敗ケース
- APIレスポンス
- 認証エラーの詳細
- 設定全体のdump
- ファイルパス
- 内部URL
- ユーザー識別子

### 6-2. よくある誤り

- デバッグのために全部出す
- 一時的なverboseログを消し忘れる
- 失敗例をそのまま共有する
- 外部の実験管理サービスに生データ断片が送られる
- Slackにログを貼る

### 6-3. なぜ危険か

MLでは、ログに残るものの価値が高い。

- 学習サンプル
- 評価対象データ
- 顧客データ
- 仕様文
- 実験条件
- モデル出力の失敗例

これらは、そのまま漏れて困る情報であることが多い。

### 6-4. 自分へのルール

- 生テキストを安易にログへ出さない
- 必要なら一部マスク・サンプリングする
- config dump は秘密を除外する
- 外部SaaSへ送られる情報を把握する
- エラー共有時は機密を削る

---

## 7. 学習データ利用: Web由来データは「公開情報」ではなく「混入データ」として扱う

### 7-1. 「公開されている」ことと「安全に使える」ことは違う

Web上にあるからといって、学習利用・再配布・長期保存・モデル化が無条件に問題ないわけではない。

### 7-2. 混入しうるもの

- 個人情報
- メールアドレス
- 電話番号
- APIキー
- 認証トークン
- 社内文書の誤公開
- 著作権上扱いが難しいコンテンツ
- 有害表現
- ノイズ
- 不正確な記述
- プロンプト注入的な文
- スクレイピングで得た規約上微妙なデータ

### 7-3. ML的に起こる問題

- モデルが丸暗記する
- evalやdemoで機密っぽい内容を吐く
- 学習データの出所説明ができない
- 顧客向け説明責任を果たせない
- 後からデータ削除依頼や利用制限に対応できない

### 7-4. 収集時に考えるべきこと

- どのソースから取ったか
- どういう条件で取得したか
- どの利用目的か
- 個人情報や秘密情報の除去方針
- ライセンス・利用規約
- データ品質基準
- 保持期間
- 削除方針

### 7-5. 自分へのルール

- Webデータは「公開だからOK」と考えない
- 学習前に機微情報混入を疑う
- データソースと処理履歴を残す
- 社内文書や顧客文書は特に厳格に分ける
- 評価用サンプルの共有も慎重にする

---

## 8. チェックポイント・モデル成果物: モデルも情報資産である

### 8-1. 見落としやすい点

モデルはコードやバイナリとして扱われがちだが、実際には以下の意味で情報資産。

- 学習データの影響を受ける
- 顧客データや社内データ由来の癖を持つ
- 再配布で外部に渡る
- 生成を通じて情報漏洩に繋がる可能性がある

### 8-2. 危険なもの

- checkpoint
- adapter
- reward model
- tokenizer
- synthetic data generator
- eval outputs
- prompt template
- instruction dataset

### 8-3. 自分へのルール

- モデル成果物を無条件に外部共有しない
- 学習元データの性質を意識する
- 社外共有可能な成果物と内部限定成果物を分ける
- demo用成果物は本番学習資産と分離する

---

## 9. チーム開発: 誰でも触れる状態を作らない

### 9-1. 危険な運用

- 共通IAMユーザー
- 共通SSH鍵
- 共通S3パス
- 誰でも本番データにアクセス可能
- 研究用と本番用が同じ場所
- 一時メンバーにも恒常権限
- 誰がやったか分からない運用

### 9-2. なぜ危険か

チームでは事故原因の多くが悪意ではなく、境界の曖昧さにある。

- 誰が何を触れるか曖昧
- 責任が曖昧
- 監査不能
- 削除事故の追跡不能
- 離職時に権限整理できない

### 9-3. 自分へのルール

- 個人識別可能な権限付与にする
- 共通鍵を常用しない
- プロジェクト・環境ごとに分離する
- 共同作業でも「誰が何をしたか」追えるようにする
- 一時参加者の権限は期限付きで考える

---

## 10. 監査と記録: 「後から追えること」が再発防止になる

### 10-1. 監査がないと何が困るか

問題が起きたときに、次が分からなくなる。

- 誰が設定を変えたか
- いつ公開状態になったか
- 誰が削除したか
- どの権限でアクセスされたか
- 想定外のリージョンやノード起動があったか

### 10-2. ML開発での典型例

- S3ポリシー変更
- IAMロール変更
- セキュリティグループ変更
- ノードの大量起動
- 推論・学習ジョブの異常実行
- 外部共有状態への変更
- 成果物の消失

### 10-3. 自分へのルール

- 重要な変更は後で追える前提で運用する
- 誰が何を変えたか残るようにする
- S3 / IAM / EC2 の変更は特に重要視する
- トラブル時は感覚ではなく記録で見る

---

## 11. コスト事故: セキュリティ事故に近いものとして扱う

### 11-1. よくあるコスト事故

- GPUインスタンスの止め忘れ
- ジョブの無限リトライ
- スポット再試行の暴走
- S3の大量出力
- EBS / snapshot の残骸
- 不要リージョンでの起動
- データ転送料の増大
- 実験ログ保存しすぎ

### 11-2. なぜセキュリティに近いか

コスト事故は、以下と根本が似ている。

- 権限が雑
- 監査が弱い
- 境界が曖昧
- 誰でも操作できる
- 一時対応が固定化

### 11-3. 自分へのルール

- 実験の終了条件を明確にする
- 不要なリソースを残さない
- 予算感覚を持つ
- 大規模実験前に保存先・出力量を見積もる
- 異常な利用を見つけられるようにする

---

## 12. 法務・倫理・説明責任: モデル開発でも避けられない

### 12-1. 気にすべき観点

- データ利用規約
- 著作権
- 個人情報
- 顧客データの扱い
- 社内情報の持ち出し
- 出力の説明責任
- データ削除要求への対応
- モデルの再配布条件

### 12-2. 特に自分の業務で意識すること

Web由来データを使う、社内データを使う、顧客データに近いものを扱う場合は、「技術的に取れるか」ではなく「使ってよいか」「将来説明できるか」を優先する。

### 12-3. 自分へのルール

- データの出所を説明できるようにする
- 顧客や社内の機微情報を学習に入れるときは特に慎重にする
- 後から問題化しそうなデータは安易に使わない
- demoや共有時に元データの性質を意識する

---

## 13. 自分が特に見落としやすいポイント

### 13-1. 学習ノードに権限を載せすぎる

「学習が動けばいい」で広くしがちだが、一番危険。

### 13-2. ログに秘密や生データを残す

モデル開発ではコードよりログ・設定・失敗例の方が漏れやすい。

### 13-3. Webデータを軽く見る

公開されていることと、安全に学習利用できることは別。

### 13-4. checkpointを安全だと思い込む

モデル成果物も情報漏洩の媒体になりうる。

### 13-5. 一時運用が常態化する

「今だけ」が半年続く、が一番危ない。

---

## 14. 今後の自分用・実践ルール集

### 14-1. 権限

- 人間と実行環境の権限を分ける
- 常設の強権限を持たない
- S3権限はprefix単位で絞る
- 削除権限は最小化する

### 14-2. データ

- 生データ・前処理後・成果物・共有物を分ける
- 派生物も機密になりうると考える
- Web由来データは混入前提で扱う
- 機微情報や個人情報の混入を疑う

### 14-3. ノード運用

- SSH前提にしすぎない
- head node に権限を集中させない
- ノードのローカル残骸を意識する
- AMI / Docker に秘密を入れない

### 14-4. Secrets

- APIキーやtokenを設定ファイルやログに残さない
- notebook共有前に秘密を確認する
- config dumpのマスキングを徹底する

### 14-5. ログ

- 生データや出力全文を安易に保存しない
- 共有前に機密を削る
- 実験管理サービスへ送る内容を把握する

### 14-6. チーム

- 共通鍵・共通ユーザーを避ける
- 誰が何をしたか追える運用にする
- 一時参加者の権限を放置しない

### 14-7. コスト

- GPUの止め忘れを防ぐ
- 実験の出力量と保存量を見積もる
- 不要なスナップショットやEBSを残さない

### 14-8. 法務・倫理

- 出所不明なデータを安易に入れない
- 使ってよいデータかを考える
- 後から説明できるようにする

---

## 15. 最後のまとめ

自分のようなML/LLMエンジニアにとっての主戦場は、典型的なWeb脆弱性というより、

- IAM
- S3
- SSH / ノード運用
- Secrets
- ログ
- 学習データ
- チーム境界
- 監査
- コスト
- 説明責任

である。

特に大事なのは、次の一文に尽きる。

**「モデル開発基盤は、巨大な計算環境であると同時に、巨大な機密情報環境でもある」**

だから今後は、
「動くか」だけでなく、
「誰が触れるか」
「何が残るか」
「どこまで漏れうるか」
「後で説明できるか」
まで含めて考える。
